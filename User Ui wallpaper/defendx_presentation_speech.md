## Slide 2 — Introduction

### Speech

In modern enterprises, cybersecurity threats do not originate only from external hackers.

A significant and growing proportion of security incidents are caused by insiders, meaning employees, contractors, or authorized users who misuse their access privileges, either intentionally or unintentionally.

According to industry reports, insider threats account for approximately 60 percent of all data breaches, and the average cost of an insider threat incident exceeds 15 million US dollars annually.

DefendX is an AI-powered desktop application designed to combat this exact problem.

It monitors real-time user behavior across Chrome browsing activity, file system access, USB connections, and VS Code workspace activity.

It runs trained Machine Learning models locally on the administrator's workstation to classify users into risk levels, and it provides an interactive Admin Portal with a built-in OS Simulator to demonstrate and test threat detection scenarios.

Additionally, the system integrates Hugging Face's Gemma-3-IT large language model as an AI forensics assistant that automatically evaluates security alerts and generates human-readable remediation advice.

---

## Slide 3 — Problem Statement

### Speech

Now let us understand the problem in detail.

Traditional cybersecurity tools such as firewalls, intrusion detection systems, and antivirus software are designed primarily to defend against external attacks.

However, they have several critical limitations when it comes to insider threats.

First, rule-based systems cannot detect novel attack patterns. An employee who slowly exfiltrates data over several weeks will not trigger any threshold-based alarms.

Second, most organizations lack visibility into Shadow AI usage. Employees may paste confidential source code or customer data into public generative AI platforms such as ChatGPT, DeepSeek, or Claude without the organization's knowledge.

Third, USB data exfiltration remains a significant risk. Employees can copy sensitive intellectual property to removable storage devices undetected.

Fourth, existing tools are heavily dependent on cloud infrastructure, which introduces latency, privacy concerns, and single points of failure.

As a result, insider threats often go undetected for months, causing massive financial and reputational damage to the organization.

## Slide 4 — Proposed Solution

### Speech

To overcome these limitations, we developed DefendX, an AI-based insider threat detection and mitigation platform.

Our system uses a dual-model machine learning pipeline.

The first model is an Isolation Forest, which is an unsupervised anomaly detection algorithm. It learns the normal behavioral baseline of all employees and flags users whose activity patterns deviate significantly from the norm.

The second model is a Random Forest Classifier, which is a supervised classification algorithm. It takes the anomaly labels generated by the Isolation Forest and trains a high-accuracy classifier that can predict whether a new user's behavior represents a genuine threat.

Both models are trained on the CERT Insider Threat Dataset version r4.2, which is a publicly available benchmark dataset provided by Carnegie Mellon University's Software Engineering Institute.

In addition to the ML pipeline, we integrated Google's Gemma-3-1b-IT instruction-tuned language model through the Hugging Face Inference API. This AI agent automatically analyzes each security alert, explains the nature of the threat, and suggests remediation actions in clear, professional language.

The entire system runs locally on the administrator's machine with no dependency on cloud services for threat detection.

## Slide 5 — Dataset Description

### Speech

Our machine learning models are trained on the CERT Insider Threat Dataset version r4.2.

This dataset was developed by the Carnegie Mellon University Software Engineering Institute specifically for research on insider threat detection.

It contains synthesized but realistic organizational activity logs for approximately 1,000 employees across multiple departments.

The dataset includes five types of behavioral logs:

One, Logon and Logoff activity, which records when employees sign in and out of their workstations, including after-hours access.

Two, HTTP web browsing logs, which track all URLs visited by employees, enabling us to detect Shadow AI usage and unauthorized cloud storage access.

Three, Device and USB connection logs, which record when removable storage devices are connected or disconnected.

Four, File access logs, which capture when employees read, copy, or transfer sensitive files.

Five, Email metadata logs, which track the number of emails sent to external addresses and the size of attachments.

From these raw logs, we engineered a feature matrix containing 15 to 20 numeric behavioral features per employee, such as after-hours logon frequency, USB device connection count, external email ratio, and Shadow AI browsing score.

## Slide 6 — System Architecture

### Speech

Let me now walk you through the system architecture of DefendX.

The architecture follows a modular, layered design with six core components.

The first layer is the System Monitor module. It continuously scans the local workstation for Chrome browsing history, VS Code workspace changes, USB device mounts, and disk access patterns using Python threading and SQLite database queries.

The second layer is the Threat Analyzer engine. It loads the pre-trained Isolation Forest and Random Forest models using Joblib deserialization and scores each user for insider threat risk.

The third layer is the Alert Management system. It maintains a severity-classified alert queue with five levels: Critical, High, Medium, Low, and Info. Each alert is timestamped, tagged with the source user ID, and routed to both the desktop notification system and the web dashboard.

The fourth layer is the AI Agent module. It connects to the Hugging Face Inference API using the Featherless-AI provider and sends each alert to the Gemma-3-1b-IT model for automated forensic analysis.

The fifth layer is the Desktop Admin Portal. Built with Python Tkinter, it provides a complete graphical interface with sidebar navigation, real-time threat dashboards with Matplotlib charts, user profile management, and PDF report generation using ReportLab.

The sixth layer is the OS Simulator Console. It spawns interactive simulated Windows and Linux desktop environments where administrators can trigger test anomalies such as USB exfiltration, Shadow AI browsing, and credential theft to validate the detection pipeline.

## Slide 7 — Technologies Used

### Speech

Now let me describe the technologies and libraries used in our project.

For the programming language, we used Python 3.11 as the primary development language for both the desktop application and the machine learning pipeline.

For machine learning, we used Scikit-learn version 1.3, which provides our Isolation Forest and Random Forest implementations. We used Pandas and NumPy for data preprocessing and feature engineering, and Matplotlib and Seaborn for generating training performance plots.

For the AI language model integration, we used the Hugging Face Hub Inference Client to connect to Google's Gemma-3-1b-IT model through the Featherless-AI provider. This model performs real-time security alert analysis through the chat completion API.

For the desktop application, we used Tkinter as the GUI framework with a custom dark-theme styling system. We used PIL and Pillow for image processing and wallpaper rendering in the OS Simulator. We used ReportLab for generating professional PDF threat assessment reports.

For real-time monitoring, we used Python's SQLite3 module to read Chrome history databases, the Watchdog library for file system event monitoring, and the Plyer library for desktop push notifications.

For packaging and deployment, we used PyInstaller to compile the entire application into a single standalone Windows executable of approximately 112 megabytes. The web download portal was built with vanilla HTML, CSS, and JavaScript.

For data serialization, we used Joblib for saving and loading trained ML models and JSON for configuration persistence.

## Slide 8 — Machine Learning Models

### Speech

Let me explain the two machine learning models used in DefendX in greater detail.

The first model is the Isolation Forest.

The Isolation Forest is an unsupervised anomaly detection algorithm that works by randomly partitioning the feature space using decision trees. Anomalous data points, which represent insider threats, are isolated in fewer partitions compared to normal data points.

We configured our Isolation Forest with 200 estimators and a contamination parameter of 0.05, meaning we expect approximately 5 percent of users to exhibit anomalous behavior. This aligns with the known insider threat ratio in the CERT dataset.

The model outputs an anomaly score for each user. Users with scores below the threshold are flagged as potential insider threats.

The second model is the Random Forest Classifier.

The Random Forest is a supervised ensemble learning algorithm that constructs 200 decision trees during training and aggregates their predictions through majority voting.

We configured it with a maximum depth of 15, a minimum sample split of 5, and balanced class weights to handle the inherent imbalance between normal and threat users.

The Random Forest was trained using the Isolation Forest's anomaly labels as pseudo-ground-truth, creating a semi-supervised learning pipeline. It achieved a strong ROC AUC score during evaluation and provides probability-based threat scores for each user.

Both models use StandardScaler normalization to ensure all features contribute equally during training.

## Slide 9 — OS Simulator Console

### Speech

One of the unique features of DefendX is the OS Simulator Console.

This module allows security administrators to simulate insider threat activities in a safe, controlled environment without affecting real systems.

The simulator provides five virtual user workstations: three running simulated Windows desktops and two running simulated Linux desktops.

Each simulated desktop includes a realistic lock screen requiring password authentication, a desktop canvas with wallpaper backgrounds scaled from actual Windows 11 and Ubuntu Linux images, and three interactive applications.

The first application is the USB Connection Simulator, which allows mounting a virtual USB drive and attempting to copy sensitive files such as source code backups and salary records. Copying restricted files triggers an immediate Critical-level alert on the Admin Portal.

The second application is the Chrome Web Browser Simulator, which includes a URL address bar and bookmark shortcuts. Navigating to GenAI platforms like ChatGPT or DeepSeek triggers a High-level Shadow AI Usage alert.

The third application is the File Explorer Simulator, which displays a virtual directory listing. Opening restricted files such as plaintext password files or SSL decryption keys triggers a credential theft or secret key leak alert.

When any simulated user triggers a threat, a real-time popup modal appears on the Admin Portal displaying the User ID, the Activity Name, and the Level of Risk.

## Slide 10 — Admin Portal Features

### Speech

The Admin Portal is the central command center for security administrators.

It starts with an elegant loading screen displaying the DefendX logo with a dynamic progress bar and cycling status messages as the system initializes its modules.

After the loading screen, administrators must authenticate using corporate credentials. In our prototype, the default credentials are admin and 12345, which will be connected to Firebase authentication in the production version.

Once authenticated, the admin portal provides nine navigation screens.

The Dashboard shows real-time threat metrics, risk distribution charts, and top threat alerts using Matplotlib visualizations embedded in the Tkinter interface.

The Threat Analysis screen provides detailed per-user risk breakdowns with explainable AI features showing which behavioral indicators contributed most to each user's threat score.

The User Profiles screen lists all monitored employees with their department, role, and computed risk level.

The Connect screen shows live connection status for Chrome, VS Code, and disk monitoring agents.

The Malware Protection screen provides file integrity monitoring and quarantine management.

The Network Packets screen captures and analyzes network traffic patterns.

The Real-Time Settings screen allows administrators to tune detection sensitivity, refresh rates, and toggle corner alert notifications.

The Reports screen generates comprehensive PDF threat assessment documents using the ReportLab library.

## Slide 12 — Results and Evaluation

### Speech

Now let me present the results of our system.

On the machine learning side, the Isolation Forest successfully identified approximately 5 percent of users as anomalous, consistent with the expected insider threat ratio in the CERT dataset.

The Random Forest classifier achieved a strong classification performance with high precision and recall on the test set, validated through ROC AUC scoring, confusion matrices, and classification reports.

Feature importance analysis revealed that after-hours logon frequency, USB device connection count, and external email ratio were the strongest indicators of insider threat behavior.

On the application side, the system successfully monitors Chrome browsing history in real time, detecting Shadow AI usage across 18 GenAI platforms and 12 cloud storage services.

The OS Simulator accurately triggers alerts across all three simulated application types: USB exfiltration, unauthorized web browsing, and sensitive file access.

The Gemma-3-1b-IT AI agent provides coherent, professional forensic analysis for each security alert within 2 to 5 seconds.

The complete application compiles into a single 112-megabyte standalone Windows executable requiring no external dependencies.

### Speech
